
# Product Requirements Document (PRD)
## One Thing That Matters – Agentic Newsletter Backend (Supabase)
### Version 1.2

---

# 1. Context and Purpose

The **One Thing That Matters** project is an agentic, partially automated newsletter system designed to help the author produce a high-quality, daily content recommendation newsletter with minimal manual effort. Each issue of the newsletter contains curated items across several domains:

- One Thing to Watch (videos)
- One Thing to Listen To (audio/podcasts)
- One Thing to Read (articles)
- (Future: “One Chart That Matters”, “One Tool That Matters”, etc.)

The long-term vision is a fully agent-assisted editorial workflow where autonomous pipelines ingest, filter, summarise, contextualise, and rank content across multiple media types. The human simply opens a UI each morning, sees a pre-ranked shortlist of high-quality candidates for each category, and selects one item per category to include in that day’s newsletter.

To enable this, each category requires:

1. Automated data ingestion  
2. Automated enrichment and reasoning (LLM pipeline)  
3. Automated scoring / ranking  
4. A centralised database that:  
   - stores all candidates  
   - prevents duplicates  
   - tracks usage state  
   - powers the selection UI  
   - supports multi-agent access  
5. A front-end where the human selects the final items for that day.

This PRD defines the backend system required to support the first vertical slice:  
**“One Thing to Watch” → The Watch Candidates Database.**

This backend will become the template for all future categories (Listen, Read, etc), enabling consistent agent workflows and a unified editorial interface.

---

# 2. Overview

The purpose of this backend is to provide a single source of truth for all YouTube video candidates generated by the n8n “One Thing That Matters – Watch” pipeline. The backend will store:

- YouTube metadata (title, video ID, publish date, etc)
- LLM-generated summary
- LLM-generated “why it matters” insight
- LLM-generated fit_score and optional fit_rationale
- Workflow state (whether the item has been selected for an issue)

The backend must support:

- Deduplicated ingestion (prevent multiple rows for the same YouTube video)
- Selection and marking of a video for a newsletter issue
- Querying top candidates for a daily shortlist
- Future expansion to “Listen” and “Read” flows

The backend will be built on Supabase Postgres, with row-level security OFF initially and a simple API key authentication model for n8n.

# 3. Core Entities

## 3.1 Video Candidate (watch_candidates)

This entity represents one unique YouTube video enriched by the n8n pipeline.

### Fields (Final Schema)

| Column | Type | Required | Notes |
|--------|------|----------|-------|
| id | uuid | yes | Primary key (Supabase default) |
| video_id | text | yes | Must be unique — YouTube video ID |
| channel_id | text | yes | Channel identifier |
| channel_name | text | yes | Channel title |
| title | text | yes | Video title |
| summary | text | yes | LLM-generated summary |
| why_it_matters | text | yes | LLM contextual explanation |
| fit_score | integer | yes | Integer 1–10 |
| fit_rationale | text | optional | Explanation for debugging |
| published_at | timestamptz | yes | YouTube published date |
| thumbnail_url | text | optional | High-res thumbnail |
| view_count | integer | optional | Metadata |
| like_count | integer | optional | Metadata |
| duration_seconds | numeric | optional | Video duration |
| url | text | yes | Full YouTube URL |
| used_in_issue_date | date | default NULL | Filled when selected |
| discarded | boolean | default false | For manual rejection |

### Constraints

- UNIQUE(video_id) enforced.  
- Primary key: id (UUID).  
- fit_score must be integer between 1 and 10.

# 4. Functional Requirements

## 4.1 Deduplicated Ingestion via n8n

- n8n POSTs or UPSERTs rows into `watch_candidates`.
- If `video_id` already exists, the row must be updated, not duplicated.

### Fields updated during upsert:
- summary  
- why_it_matters  
- fit_score  
- fit_rationale  
- view_count  
- like_count  
- published_at  
- title  
- thumbnail_url  

### Fields NOT updated:
- used_in_issue_date  
- discarded  

---

## 4.2 Candidate Shortlist Query

```
SELECT *
FROM watch_candidates
WHERE used_in_issue_date IS NULL
AND discarded = false
ORDER BY fit_score DESC, published_at DESC
LIMIT 5;
```

---

## 4.3 Selection for Newsletter Issue

When the user picks a video:

```
used_in_issue_date = today
```

This permanently removes it from future shortlists.

---

## 4.4 Discarding Candidates

```
discarded = true
```

Discarded candidates never appear in shortlists.

# 5. API Requirements

Supabase REST auto-generates these endpoints:

## POST /watch_candidates
- Inserts or upserts candidate.
- Must support `on_conflict=video_id`.

## GET /watch_candidates
Supports filters:
- used_in_issue_date=is.null  
- discarded=is.false  
- order=fit_score.desc  

## PATCH /watch_candidates?id=eq.{id}
- Update used_in_issue_date  
- Update discarded  

---

# 6. Security Requirements

- RLS disabled initially.
- n8n uses service-level API key.
- Frontend uses anon key with read-only policy:
  - Cannot modify or delete rows.
  - Cannot set used_in_issue_date.

---

# 7. Migration / State Considerations

- Google Sheet rows will be migrated manually via n8n.
- Supabase becomes the single source of truth.
- Sheets remain optional debug tools.

---

# 8. Success Criteria

- No duplicate videos in DB.
- Daily shortlist consistently returns 5 highest-fit candidates.
- No previously selected video ever appears again.
- Schema can expand to Listen, Read, and other categories with minimal changes.
